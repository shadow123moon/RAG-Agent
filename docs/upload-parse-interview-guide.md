# PaiSmart 文档上传与解析流程面试讲解版

这篇文档只讲一件事：在 `PaiSmart` 这个项目里，用户上传一个文档之后，系统到底是怎么一步一步把它变成“可检索、可问答”的知识库内容的。

我会尽量用面试里能讲得出来的话来解释，而不是把源码逐行翻译一遍。你看完这篇文档，目标不是立刻能改完整个系统，而是至少能回答下面这些问题：

- 文档为什么要分片上传？
- 上传成功之后，文件先去了哪里？
- 为什么还要走 `Kafka`，不能直接在接口里解析吗？
- 文档解析到底做了什么？是不是就是简单调个库？
- 切片为什么要切？怎么切？
- 最终为什么又写数据库，又写 Elasticsearch？
- 聊天问答为什么能够命中这份文档？

如果把这条链路压缩成一句话，它其实是：

`前端分片上传 -> 后端落分片到 MinIO 并记录状态 -> 合并成完整文件 -> 估算解析/向量化成本 -> 发 Kafka 异步任务 -> 消费者下载完整文件 -> 解析正文并切片 -> 先把切片写数据库 -> 调 Embedding 生成向量 -> 批量写入 Elasticsearch -> 聊天时按问题检索这些切片`

下面开始按过程讲。

---

## 1. 为什么这个项目要把“上传”和“解析”拆开

第一次看这个项目时，最容易误会的一点是：以为“上传文档”就是一个接口把文件收了，然后顺手解析一下，最后存进去就完了。

但真实工程里不是这样。

文档上传和文档解析虽然是同一条业务链上的两个阶段，但它们的压力模型完全不同。

上传阶段的特点是：

- 用户在前端点上传后，希望接口尽快有响应；
- 文件可能比较大，网络中断、重复上传、续传都要考虑；
- 系统主要关心的是“这个文件有没有完整、安全地到达服务端存储”。

解析阶段的特点是：

- 速度不一定快，尤其是 `PDF`、`Word`、超长文本；
- 解析后还要切片、估算 token、调外部 embedding 模型、写入 `Elasticsearch`；
- 任何一步失败，都可能导致整个流程失败；
- 这是一个典型的重任务，不适合直接阻塞上传接口。

所以这个项目的设计思路是：

1. 先把文件可靠地收上来；
2. 再把“处理这个文件”的任务异步化；
3. 由后台消费者慢慢做解析和向量化。

这也是为什么它会引入：

- `MinIO`：用来存文件；
- `Redis`：用来记分片上传状态；
- `MySQL`：用来记文件元数据和切片文本；
- `Kafka`：把上传和解析解耦；
- `Elasticsearch`：存最终用于检索的向量化切片。

你可以把它理解成两条子链路：

- 第一条是“把文件收上来”；
- 第二条是“把文件变成知识库可检索内容”。

---

## 2. 第一步：前端为什么不是整文件上传，而是分片上传

这个项目的上传入口在 `UploadController`，接口路径是 `/api/v1/upload/chunk`。从名字就能看出来，它不是直接传整文件，而是按分片上传。

这样设计的原因很现实：

- 大文件一次上传容易超时；
- 网络中断后，如果重新传整文件，代价很大；
- 分片后可以断点续传；
- 服务端可以知道哪些片已经到了，哪些还没到。

这个接口在接收每一个分片时，通常会收到这些信息：

- `fileMd5`：整文件的唯一标识；
- `chunkIndex`：当前是第几个分片；
- `totalSize`：整文件总大小；
- `fileName`：文件名；
- `totalChunks`：总分片数；
- `orgTag`：组织标签；
- `isPublic`：是否公开；
- `file`：当前这一片的真实二进制内容。

注意这里最重要的字段其实不是文件名，而是 `fileMd5`。

为什么？

因为文件名是不可靠的，同名文件很多，而 `MD5` 更适合作为整条上传链路里的“稳定标识”。后面数据库记录、分片路径、合并后的对象路径，基本都围绕这个值来组织。

---

## 3. 第二步：后端收到分片后，先做哪些校验

在真正保存分片之前，控制层会先做几类判断。

### 3.1 文件类型校验

项目里只在第一个分片到达时做文件类型校验。这个思路很合理，因为文件名和扩展名在整次上传过程中不会变，没必要每个分片都重复校验一次。

这里校验的目的不是“这个文件能不能存”，而是“这个文件后续能不能被系统解析并进入知识库”。

也就是说，上传接口不仅是一个存储入口，它还带着业务语义：系统只想接收那些后面能进入知识库处理链路的文档类型。

### 3.2 组织和权限校验

如果请求里没传 `orgTag`，系统会去查当前用户的主组织标签。因为这个项目不是一个纯单用户知识库，它带有组织维度和可见性控制，后面文档解析完以后，这些权限信息还要继续往下传。

### 3.3 文件大小限制

项目里还做了组织级别的上传大小限制。对非管理员用户，如果文件太大，会在上传阶段就被拒绝。

这一步的价值在于：尽可能早失败。不要让一个本来就不允许上传的大文件，把分片都传完、合并完、甚至开始解析以后才报错。

---

## 4. 第三步：真正保存分片时，系统做了哪三件事

这一段是上传服务 `UploadService` 的核心。可以把它理解成：每收到一个分片，系统就会同时在三个维度上做记录。

### 4.1 在 MySQL 里建立或查询文件主记录

项目里有一个 `file_upload` 表，对应 `FileUpload` 实体。

当某个文件的第一个分片到来时，系统会先看这个 `fileMd5 + userId` 的文件记录是不是已经存在：

- 如果不存在，就创建一条新的文件主记录；
- 如果存在，就复用这条记录。

这里的状态值很值得记一下：

- `0`：上传中
- `2`：合并中
- `1`：已完成

但这里要特别注意一个容易误解的点：

**这个“已完成”在当前实现里更接近“文件合并完成”，而不完全等于“知识库解析与向量化也全部完成”。**

这点你在面试里如果能点出来，会显得你不是只会背流程，而是真的看过实现。

### 4.2 把当前分片存到 MinIO

分片真正的二进制内容不会直接进数据库，而是存到 `MinIO`，路径大概是：

`chunks/{fileMd5}/{chunkIndex}`

这样设计的好处是：

- 对象存储更适合放二进制文件；
- 路径天然有层级结构；
- 用 `fileMd5 + chunkIndex` 可以很稳定地定位一个分片；
- 后面合并时只要按顺序把这些对象拼起来就行。

### 4.3 在 Redis 里把这个分片标记为“已上传”

项目没有用一个关系表去实时记录“某个文件哪些分片到了”，而是用了 `Redis bitmap`。

键大概长这样：

`upload:{userId}:{fileMd5}`

然后用 `chunkIndex` 对应的位去标记当前分片是否已经上传。

这样做的好处非常明显：

- 节省空间；
- 判断某一片有没有上传非常快；
- 获取整体上传进度也方便；
- 天然适合断点续传场景。

### 4.4 在数据库里保存分片元数据

除了 Redis 里的位图状态，项目还会把每个分片的元信息落进 `chunk_info` 表，包括：

- `fileMd5`
- `chunkIndex`
- `chunkMd5`
- `storagePath`

这里可以把 Redis 和 MySQL 的职责分开理解：

- Redis 负责快读快写的“状态标记”；
- MySQL 负责更稳定、可追溯的“分片元数据”。

这也是一个挺典型的工程思路：热状态放缓存，关键元数据落数据库。

---

## 5. 第四步：上传阶段怎么支持断点续传和幂等

这个项目在上传上有一个不错的点，就是它明显考虑了幂等和补偿。

比如某个分片如果已经在 Redis 里被标记为上传过，系统不会无脑再传一次。它还会进一步检查：

- 数据库里有没有这条分片记录；
- `MinIO` 里这个对象是不是真的存在。

如果 Redis 说“传过了”，但数据库里没记录，或者对象存储里也没有这个分片，那么系统会把它视为状态不一致，再走补偿逻辑。

这说明作者意识到了：上传链路里可能出现部分成功、部分失败的情况。不能只相信一个地方的状态。

从面试表达上，你可以把这一段总结成：

**上传服务通过 Redis bitmap、分片表、对象存储三者交叉校验，尽量保证分片上传的幂等性和状态一致性。**

---

## 6. 第五步：所有分片传完后，为什么还要单独调用“合并”

上传分片只是说明“零件都到了”，但系统真正需要的是一个完整文件。于是前端会调用 `/api/v1/upload/merge` 这个接口，请后端把这些分片拼回去。

这一步不是简单的字符串拼接，而是整个上传阶段的收口点。

在合并前，后端会做这些事情：

1. 查文件主记录，确认文件确实属于当前用户；
2. 检查状态，如果已经在合并中或者已经合并完成，就做冲突处理或幂等返回；
3. 核对分片数量是不是齐了，如果没齐，直接拒绝合并；
4. 用数据库里的分片列表按顺序读取所有 `MinIO` 分片对象。

真正的合并动作，是调用 `MinIO` 的 `composeObject` 能力，把多个对象按顺序组合成一个完整对象。

合并完成后，系统会把完整文件存到：

`merged/{fileMd5}`

然后做几件收尾动作：

- 删除原来的分片对象；
- 删除 Redis 里的分片位图标记；
- 更新文件主记录状态；
- 生成完整文件的预签名访问 URL。

这里你可以看出一个很重要的工程思想：

**分片只是中间态，完整文件才是后续解析和预览真正依赖的主对象。**

---

## 7. 第六步：为什么合并成功后，系统还会先“估算”一次成本

在 `merge` 接口里，有一个很容易被忽略但其实很业务化的动作：系统会在发 Kafka 任务之前，先对完整文件做一次 `Embedding` 用量估算。

这个估算不是为了真的开始向量化，而是为了提前知道：

- 这份文档大概会被切成多少块；
- 大概要消耗多少 embedding token。

为什么要提前算？

因为这个项目不是一个完全无限制的本地工具，它带有 token 配额和用量统计逻辑。提前估算可以用于：

- 前端展示；
- 配额判断；
- 后台统计；
- 给用户或管理员预期。

这里的一个细节是：**估算发生在解析之前，但并不直接写 ES。**

它更像是“正式处理前的一次预算”。

---

## 8. 第七步：为什么合并完不直接解析，而是发 Kafka

这是整条链路里的一个关键设计点。

在 `merge` 成功后，控制层会构造一个 `FileProcessingTask`，里面至少包含：

- `fileMd5`
- `filePath`
- `fileName`
- `userId`
- `orgTag`
- `isPublic`

然后通过 `KafkaTemplate.executeInTransaction` 把这个任务发到 Kafka 主题里。

注意这里用了事务发送，这至少表达了两个意图：

1. 希望任务投递更可靠；
2. 上传合并阶段和异步处理阶段的衔接，最好不要随便丢消息。

为什么一定要走 Kafka？

因为如果不走 Kafka，而是在 `merge` 接口里直接继续做：

- 下载完整文件
- 调解析库
- 切片
- 调 embedding 模型
- 写 ES

那么这个接口会非常重，容易：

- 超时；
- 阻塞；
- 失败影响用户体验；
- 一旦外部模型响应慢，整个接口就挂住。

所以这里 Kafka 的角色非常明确：

**它不是业务存储，而是异步任务总线。它把“用户上传”这件事和“后台处理文档”这件事解耦了。**

---

## 9. 第八步：Kafka 消费者拿到任务后，先做什么

异步消费者是 `FileProcessingConsumer`。

它收到任务后，会先根据 `filePath` 去下载完整文件。这里的 `filePath` 往往是一个预签名 URL，也可能是本地路径。消费者内部会统一把它转换成一个 `InputStream`。

然后它做两件大事：

1. 调 `parseService.parseAndSave(...)`
2. 调 `vectorizationService.vectorizeWithUsage(...)`

也就是说，在这个项目里，“解析”和“向量化”是两步，不是一口气混在一起做的。

这点非常关键。

很多人以为知识库系统里“解析文档”就是直接调 embedding 模型把整篇内容变向量。其实不是。

正确顺序应该是：

`先把文档里的正文抽出来 -> 再按合适粒度切成块 -> 再对这些块做向量化`

---

## 10. 第九步：文档解析到底做了什么

这一段是你前面特别关心的问题：是不是“调个库就自动切块”？

答案是：

**底层解析能力确实依赖成熟库，但这个项目在“如何切、如何清洗、如何保存中间结果”上，还是做了不少工作。**

### 10.1 先区分 PDF 和非 PDF

项目会先判断文件头是不是 `%PDF-`。

- 如果是 PDF，就走 PDF 专门逻辑；
- 如果不是 PDF，就走通用文档解析逻辑。

这样做很合理，因为 PDF 在知识库场景里通常最麻烦，单独处理能拿到更多结构信息，比如页码。

### 10.2 非 PDF：用 Tika 做通用解析，但不是一次性全读进内存

对一般文档，项目用的是 `Tika AutoDetectParser`。

但这里不是简单地“parse 一下拿整篇字符串”，而是做了一个流式 `StreamingContentHandler`。它内部维护一个较大的缓冲区，达到所谓“父块”大小后，就会触发一次处理。

为什么这样设计？

因为如果文档很大，一次性把整个正文读进内存风险很高，容易内存飙升甚至 `OOM`。所以项目在解析前还会检查当前内存使用率，如果超过阈值，还会先尝试触发 `GC`。

这说明解析模块考虑的不只是“能不能把字抽出来”，还考虑了“大文件场景下能不能稳地跑完”。

### 10.3 PDF：用 PDFBox 按页抽取文本

对 PDF，项目不是一股脑整本抽，而是逐页提取文本。这样做的好处有两个：

1. 页码信息天然保留下来了；
2. 后面做引用展示时，更容易告诉用户答案来自第几页。

更有意思的是，它还做了 PDF 页眉页脚清洗。

具体思路是：

- 先扫描每一页顶部和底部的几行；
- 统计哪些边界行在多页里反复出现；
- 如果某一行在很多页都重复，就把它视为页眉或页脚噪声；
- 在正式文本里把这些重复边界行移除。

这一步的意义非常大。

因为 PDF 文档里最烦人的不是“正文抽不出来”，而是抽出来以后混着一堆重复页码、页脚、版权说明、章节头。这些噪声如果不清掉，后面切片和检索效果都会明显变差。

所以从知识库效果上讲，这一步并不简单，也不是纯调库自动完成的。

---

## 11. 第十步：为什么还要切片，不能整篇文档直接向量化

这也是知识库系统里的核心问题。

如果把整篇文档直接向量化，会有几个问题：

- 文档太长，超出模型输入限制；
- 一个向量承载的信息太杂，检索不精准；
- 用户问的是一个小问题，但你返回的是整篇内容，噪声太大；
- 引用定位困难，不知道具体命中的哪一段。

所以项目会把解析出来的正文继续拆成多个更小的语义块。

在这个项目里，切片策略大概是分层做的：

1. 先按段落切；
2. 如果一个段落太长，再按句子切；
3. 如果一个句子还太长，再用 `HanLP` 做分词级切分；
4. 如果分词异常，再退化为按字符长度硬切。

这套策略的核心思想是：

**尽量保持语义完整，但又不能让块过长。**

换句话说，它不是简单的“每 512 个字符切一刀”，而是尽量沿着更自然的边界切。

这就是为什么项目里把这一步命名成“带语义的切块”。

---

## 12. 第十一步：解析结果为什么先写 MySQL，而不是直接写 Elasticsearch

这也是一个很值得讲的设计点。

项目里解析完一个切片后，会先把它写进 `document_vectors` 这张表，对应 `DocumentVector` 实体。这里保存的是：

- `fileMd5`
- `chunkId`
- `textContent`
- `pageNumber`
- `anchorText`
- `userId`
- `orgTag`
- `isPublic`

也就是说，**数据库里先保存的是“切好的文本块和元数据”，不是最终向量检索结果。**

为什么要这样做？

因为这样可以把“解析”与“向量化”进一步拆开：

- 解析成功但向量化失败时，中间文本结果还在；
- 重建索引时，不一定非得重新上传文件；
- 可以先检查切片是否正常，再看向量化问题；
- 有利于排障和补偿。

你这次实际排问题时就已经感受到这个设计的价值了：文档切片已经进了数据库，但 ES 里没有文档，说明问题出在向量化或索引阶段，而不是前面的上传和解析阶段。

---

## 13. 第十二步：向量化阶段真正做了什么

向量化服务 `VectorizationService` 会从数据库把刚才那些切片重新取出来，然后把每个切片的 `textContent` 提交给 `EmbeddingClient`。

返回结果是每个切片对应的一组浮点向量。

接下来系统会把文本块和向量组装成 `EsDocument`，字段里会带上：

- 文本内容；
- `chunkId`；
- 页码；
- 锚点文本；
- 向量；
- 模型版本；
- 用户和组织权限信息。

最后通过 `ElasticsearchService.bulkIndex(...)` 批量写入 `knowledge_base` 索引。

从这一刻开始，这份文档才真正进入了“可检索状态”。

所以你可以把整个处理链路理解成两次落库：

第一次落库：

- 落 `MySQL document_vectors`
- 目标是保存切片文本和中间结果

第二次落库：

- 落 `Elasticsearch knowledge_base`
- 目标是支持向量检索和后续问答

---

## 14. 第十三步：聊天为什么最终能命中这份文档

当用户在聊天里提问时，系统不会再去原始 PDF 里找答案，也不会去 MinIO 里临时读文件。

真正参与检索的是前面已经写进 `Elasticsearch` 的那些切片文档。

也就是说，聊天链路依赖的不是“上传过文件”这个事实，而是“该文件已经被成功解析、切片、向量化并进入索引”。

这也是为什么你前面测试时会出现这种现象：

- 文件明明上传了；
- 前端也能看到它；
- 但聊天里就是答不出来。

因为“上传成功”和“进入知识库可检索状态”不是一回事。

中间只要有一步失败，比如：

- Kafka 没消费；
- 解析失败；
- token 配额不足；
- embedding 维度和 ES 索引维度不一致；
- bulk index 失败；

都会导致文件停留在“看起来已经上传了，但问答检索不到”的状态。

这也是这个项目真正值得讲的地方：**知识库项目最难的往往不是调用大模型，而是把中间这些工程链路全部打通。**

---

## 15. 第十四步：这条链路里最容易出问题的地方有哪些

如果你是面试讲解，最好不要只说“流程是这样的”，还要能讲两个风险点。这个项目里至少有这些高频风险点：

### 15.1 上传阶段

- 文件类型不支持；
- 组织上传大小超限；
- 分片重复上传或状态不一致；
- `MinIO` 上传失败；
- 分片数量不全却尝试合并。

### 15.2 合并阶段

- 文件已经在合并中；
- 合并后的对象路径不存在；
- 预签名 URL 过期；
- 分片对象缺失。

### 15.3 解析阶段

- 大文件内存占用高；
- PDF 里有大量页眉页脚噪声；
- 文档正文抽取质量不稳定；
- 切片策略不合理导致块太碎或太大。

### 15.4 向量化与入索引阶段

- embedding token 配额不足；
- 外部 embedding 接口失败；
- 向量维度与 ES 索引映射不一致；
- ES 写入失败。

你前面实际踩中的两个问题就特别典型：

1. `Embedding` 模型返回 `1024` 维，但 ES 索引按 `2048` 维建的，导致解析成功、切片成功，但写索引失败；
2. 重建索引时又被 embedding token 余额限制拦住，导致消费者流程走不完。

这两类问题很适合拿来做面试里的“排障案例”。

## 16. 如果面试官让我用三分钟讲这条链路，我怎么讲

你可以直接按下面这个口径讲：

“这个项目的文档上传和解析不是一个同步接口一口气做完的，而是拆成了上传链路和异步处理链路两部分。前端先做分片上传，后端在上传接口里完成文件类型、权限和大小校验，把每个分片存到 MinIO，同时用 Redis bitmap 记录分片上传状态，并在数据库里保存分片元信息。等所有分片上传完成后，再调用合并接口，把分片在 MinIO 里合成为完整文件，生成预签名 URL，并先估算文档大概会产生多少切片和多少 embedding token。接着后端把文件处理任务发到 Kafka，由消费者异步下载完整文件并进行解析。解析时如果是普通文档，走 Tika 的流式解析；如果是 PDF，则按页提取文本，并清理重复页眉页脚。解析后的正文不会整篇直接入库，而是按段落、句子以及 HanLP 分词做语义切片，先保存到数据库表里。之后再把这些切片送到 embedding 模型生成向量，最终批量写入 Elasticsearch 的 knowledge_base 索引。聊天检索时命中的其实就是这些切片文档，而不是原始文件本身。” 

这段你如果能自己复述顺，已经够应对大多数“一面项目讲解”了。

## 17. 最后做个总总结

这个项目里，文档上传和解析的本质不是“调个库把文件读出来”，而是一条分阶段的工程链路。

它大致可以分成下面九个动作：

1. 前端按分片上传文件；
2. 后端校验类型、权限和大小；
3. 分片写入 MinIO；
4. Redis 记录分片上传状态，MySQL 记录分片元数据；
5. 所有分片传完后合并成完整文件；
6. 估算解析和向量化成本；
7. 把处理任务发到 Kafka；
8. 消费者异步解析正文并做语义切片，先写数据库；
9. 调 embedding 生成向量并批量写入 Elasticsearch。

如果你只记一句话，那就记：

**上传解决的是“文件可靠到达系统”，解析解决的是“文件变成可检索知识”，而 Kafka、MinIO、MySQL、Redis、Elasticsearch 都是在为这两件事分工协作。**

如果你把这句话真正理解了，再回头看源码时就不会再觉得它是一团乱麻，而会知道每个组件在这条链路里到底负责哪一段。
